# Сигналы изменений в агентных системах

## Контекст
**Где:** Экосистема разработки (IDEs) и OpenHands (ранее OpenDevin).
**Событие:** Внедрение Agent Client Protocol (ACP) и разделение задач на «Inner Loop» (IDE) и «Outer Loop» (Cloud).

## Наблюдение
1. **Стандартизация взаимодействия (ACP):** Появился протокол ACP (Agent Client Protocol) — аналог LSP, но для AI-агентов. Он позволяет редакторам (Zed, JetBrains, Neovim) подключаться к любым ACP-совместимым агентам через стандартный JSON-RPC интерфейс, устраняя необходимость в кастомных плагинах для каждой пары «IDE — Агент».
2. **Архитектурное разделение «Loops»:** Формализовано разделение агентных задач на:
    *   *Inner Loop (Локально/IDE):* Ad-hoc задачи, помощь в написании кода, требует подтверждения действий.
    *   *Outer Loop (Облако/CI):* Масштабируемые, автономные задачи (рефакторинг, устранение CVE, анализ логов), выполняемые флотилиями (fleets) изолированных агентов без прямого участия человека.

## Причина
*   **Фрагментация инструментов:** Разработчики привязаны к любимым IDE, а создатели агентов вынуждены писать интеграции под каждый редактор.
*   **Масштабируемость и безопасность:** Локальные агенты ограничены ресурсами машины разработчика и рисками безопасности (доступ к локальной ФС). Облачные агенты («Outer Loop») позволяют запускать тысячи изолированных инстансов для массовых задач (например, исправление уязвимости во всех репозиториях компании) с меньшим "blast radius".

## Следствие
*   **Унификация экосистемы:** Появление стандарта (ACP) снизит порог входа для новых агентных инструментов, так как им достаточно поддержать один протокол для доступа ко всем популярным IDE.
*   **Сдвиг в автоматизации:** Перенос рутинных задач (Tech Debt, CVE, логи) из зоны ответственности разработчика (Inner Loop) в зону ответственности автономных облачных агентов (Outer Loop), работающих параллельно с командой.
